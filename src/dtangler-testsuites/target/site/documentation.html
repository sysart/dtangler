                                                                                      dtangler 1.1.1

                                                                                      dtangler is a tool for analysing dependencies and producing DSMs (Dependency Structure Matrices). Currently, dtangler supports the analysis of Java classes.

                                                                                      With dtangler, you can define project-specific design rules that specify what dependencies are allowed, and what aren't. This encourages SW architects to consider and communicate the dependency structure in detail, and helps identify any possible deviations from the defined rules.

                                                                                      dtangler comes with a command line UI and a GUI. The command line version is mostly useful when integrating dtangler with automated builds. Its return value tells whether or not the analysed input contains violations. The GUI version can be used to examine DSMs in various detail levels.

                                                                                      The about dependencies page has more background information on dependencies and DSMs; the purpose of this document is to describe how to use dtangler.

                                                                                      Command line tool

                                                                                      This section explains how to use the command line version of dtangler. You can either provide your configuration straight from the command line or from a properties file. Command line parameters override any parameters of the same type given in the properties file.

                                                                                      dtangler outputs a DSM of the dependencies and reports all encountered violations. The exit value is non-zero if violations were found.

                                                                                      dtangler is run from the command line by entering: 

                                                                                      java -jar dtangler-core.jar <list of run options>

                                                                                      Quick start

                                                                                      Summary of run options 

                                                                                      Usage: 
	-<run option>=<argument>

	You can enter multiple arguments to the same option by separating
	them with a semicolon (;).

	You can enter multiple values to the same argument by separating
	them with a comma (,).

List of possible run options:

Setup: 

	-configFile=<path and name of the configuration properties file>
		Path of the config file.

	-input=<path to folder containing the items being analyzed>
		Path of items being analyzed.
		-input= is interchangeable with the older -classPath= run option.

	-ignoreFileMask=<mask for items ignored>
		Items being ignored.

Configuring dependency rules: 

	-cyclesAllowed=<true|false>
		Allow/disallow cyclic dependencies. Default value: false

	-groups=<group name> contains <item name> does not contain <item name>
		Group items to simplify configuration.
		-groups= is interchangeable with the older -group= run option.

	-rules=<item name> cannot depend on <item name>;<item name> can depend on <item name>
		Specify forbidden and allowed dependencies. 
		Allowed dependencies override forbidden dependencies.

Java-specific parameters: 

	-scope=<locations|packages|classes>
		Specify the scope of the dependency analysis. 
		Default value: packages

The full documentation with examples can be read from:
	http://www.dtangler.org/documentation

                                                                                      Run options: Setup

                                                                                      Config file

                                                                                      You can specify a properties file path if you want to use a previously created configuration.

                                                                                      For example: 

                                                                                      -configFile=/C:/Documents and Settings/kaisakit/DSM/svn/dtangler/dtangler-core/_build/eclipse/classes_test/org/dtangler/core/acceptancetests/input/exampleConfig.properties

                                                                                      Config file parameters are similar to the command line parameters, with the following exceptions:

                                                                                      • The leading '-' from each parameter is omitted
                                                                                      • Double quotes from all paths are omitted
                                                                                      • Each run option is specified on its own row
                                                                                      • You can divide long parameters to multiple rows by adding '\' to the end of the row being divided

                                                                                      The contents of an example config file: 

                                                                                      cyclesAllowed=true

groups=util contains *util*

rules=@util cannot depend on org.app; \
      org.ui cannot depend org.gui

                                                                                      Command line parameters override config file parameters

                                                                                      If the same run option is defined in both the command line and properties file, the command line version is used.

                                                                                      Analysis path

                                                                                      input defines the path to folder(s) containing the items for the dependency analysis. You can enter multiple paths by separating them with semicolons (;).

                                                                                      For example: 

                                                                                      -input=c:/projects/myProgram/build;temp/stuff

                                                                                      If the path contains empty spaces, remember to surround the path with double quotes (e.g. "C:/Documents and Settings").

                                                                                      Backward compatibility: the older classPath run option is interchangeable with input.

                                                                                      Ignore files from the analysis

                                                                                      The ignoreFileMask run option lets you ignore files from the dependency analysis.

                                                                                      • you can enter multiple masks with semicolons
                                                                                      • you can use asterisks (*) as wildcards

                                                                                      For example, if you want to exclude test files from the analysis, you could specify: 

                                                                                      -ignoreFileMask=*test*;*Test*

                                                                                      Run options: Configuration

                                                                                      Forbidden dependencies

                                                                                      Specifies what dependencies are not allowed.

                                                                                      Usage: 

                                                                                      -rules=<item name> cannot depend on <item name>
                                                                                      • You can enter multiple items by separating them with commas (,).
                                                                                      • You can enter multiple rules by separating them with semi-colons (;).

                                                                                      For example: 

                                                                                      -rules=y cannot depend on x, z; b, c cannot depend on a

                                                                                      Cycles are disallowed by default

                                                                                      If the cyclesAllowed option is omitted, cycles are treated as errors.

                                                                                      Specify groups to simplify rules

                                                                                      Configuring dependency rules that apply to multiple items is easier when done with groups. You can also use asterisks (*) as wildcards when entering group members.

                                                                                      Usage: 

                                                                                      -groups=<group name> contains <member name>
                                                                                      • You can enter multiple members to a group by separating them with commas.
                                                                                      • You can enter multiple groups by separating them with semi-colons.

                                                                                      For example: 

                                                                                      -groups=MyGroup contains foo, bar; PublicGroup contains *public*

                                                                                      Backward compatibility: the older group run option is interchangeable with groups.

                                                                                      Exclude items from groups

                                                                                      Items can be excluded from a group with the does not contain option.

                                                                                      For example: 

                                                                                      -groups=Org contains org.* does not contain org.public.*,org.util.*

                                                                                      Allowed dependencies

                                                                                      Allowed dependencies provide a way to make exceptions to forbidden dependencies. Allowed dependencies override forbidden dependencies.

                                                                                      Allowed dependencies are listed alongside forbidden dependencies in rules, with the parameter can depend on.

                                                                                      Consider the following example: 

                                                                                      -groups=Domain contains org.domain.*
-rules=@Domain cannot depend on @Domain; org.domain.public.foo can depend on org.domain.public.bar
                                                                                      • A group that contains all items starting with 'org.domain.' is specified.
                                                                                      • All dependencies between the items in the group are forbidden.
                                                                                      • The 'can depend on' rule overrides the 'cannot depend on' rule and allows dependencies from org.domain.public.foo to org.domain.public.bar.

                                                                                      Run options: Java-specific

                                                                                      With scopes, you can select the level of detail for the dependency analysis. For example, with package scope, Java packages will be used as rows and columns in the DSM.

                                                                                      Default scope

                                                                                      If the scope option is omitted, package scope is used.

                                                                                      Location scope

                                                                                      Level of detail: the locations for the class files being analysed, given with the input run option.

                                                                                      Analyzing dependencies between locations can come in handy if you want to see how projects depend on each other, for example.

                                                                                      Package scope

                                                                                      Java package level detail.

                                                                                      Class scope

                                                                                      Java class level detail.

                                                                                      Return values and error messages

                                                                                      If any of the specified rules are broken in the analysed code, dtangler will produce a rule violation message and exit with a non-zero value.

                                                                                      Cycles are always reported. They cause a non-zero exit value by default.

                                                                                      Cycles found when cycles are allowed

                                                                                      If cycles are allowed, cycles are reported but they do not cause an error.

                                                                                      For example, a possible output could be: 

                                                                                      Cycle: Dependable1-->Dependable2-->Dependable1

                                                                                      Cycles found when cycles are disallowed

                                                                                      Cycles are reported, and dtangler's exits with a non-zero value.

                                                                                      No cycles found

                                                                                      If cycles are not found, no cycle violations are reported and the exit value is not affected.

                                                                                      No rule violations

                                                                                      If no rule violations are found, the exit value is not affected.

                                                                                      Rule violations between single items

                                                                                      When a forbidden dependency has been found, the broken rule is reported in the output.

                                                                                      For example, if the rule 

                                                                                      -rules=A cannot depend on B

                                                                                      is broken, the following violation is produced: 

                                                                                      Rule violation: A cannot depend on B

                                                                                      Rule violations between groups

                                                                                      Rules constructed with groups are handled in the same way as rules with single items.

                                                                                      Consider the following arguments: 

                                                                                      -groups=Util contains *util*;App contains *app*
-rules=@Util cannot depend on @App

                                                                                      If there is a dependency between any members of the groups, the output will be: 

                                                                                      Rule violation: Group Util cannot depend on Group App

                                                                                      Rule violations and excluded items

                                                                                      Dependencies to and from items that were excluded from groups do not affect the outcome of group rules.

                                                                                      For example, with dependencies A->C->B, the following arguments won't generate a violation because C has been excluded from the group: 

                                                                                      -groups=All contains * does not contain C
-rules=@All cannot depend on @All

                                                                                      Using groups and single items together

                                                                                      You can combine single items and group items freely in rule definitions: 

                                                                                      -groups=Foo contains eg.foo*; Bay contains eg.bay*
-rules=@Foo cannot depend on eg.bar; eg.bar cannot depend on @Bay

                                                                                      With dependencies eg.foo->eg.bar->eg.bay, the resulting violation output would look like this: 

                                                                                      Rule violation: eg.bar cannot depend on Group Bay
Rule violation: Group Foo cannot depend on eg.bar

                                                                                      Violations on lower scope

                                                                                      A notification is printed if an item contains violations on any of its more detailed scopes.

                                                                                      For example, a cycle between two classes, Cat and Dog, both inside the same Java package animals, will produce a notification of this type on package scope: 

                                                                                      animals contains a violation: Cycle: Cat-->Dog-->Cat

                                                                                      GUI

                                                                                      The dtangler GUI is an interactive, graphical standalone tool to help you analyze dependencies. The GUI can also be used to create configurations that can be used with the command line dtangler to automatically analyze dependencies. This is particularly useful when this is done through continuous integration.

                                                                                      Basic features

                                                                                      View information about dtangler

                                                                                      Additional information and the version number of dtangler can be viewed by selecting 'About' from the 'Help' menu.

                                                                                      Exit the gui

                                                                                      The dtangler GUI can be closed by selecting 'Exit' from the 'File' menu.

                                                                                      Managing configurations

                                                                                      The configurations that you make inside the Gui can be saved to- and loaded from files.

                                                                                      The dtangler gui configuration files are compatible with the ones used by the dtangler command line version.

                                                                                      Current configuration file name

                                                                                      The name of the currently open configuration file can be seen in the title bar behind the Dtangler DsmUI title.

                                                                                      If the configuration has never been saved, the titlebar will display unsaved settings.

                                                                                      If the configuration has been change since it was last saved or created, a '*' will be displayed in front of the configuration filename.

                                                                                      Creating a new configuration

                                                                                      A new configuration can be created by selecting 'New' from the 'File' menu or by pressing Ctrl+N.
                                                                                      This is essentially the same as resetting all configuration parameters back to their defaults.

                                                                                      Changed configuration upon creating a new configuration

                                                                                      If the current configuration contains changes since the last time it has been saved, you will be asked whether to save those changes or not before the new configuration is created.

                                                                                      Open an existing configuration file

                                                                                      An existing configuration can be opened by selecting 'Open' from the 'File' menu or by pressing Ctrl+O.

                                                                                      Changed configuration upon opening an existing configuration file

                                                                                      If the current configuration contains changes since the last time it has been saved, you will be asked whether to save those changes or not before another configuration is opened.

                                                                                      Save configuration

                                                                                      At any time you can save the current configuration by selecting 'Save' from the 'File' menu or by pressing CTRL+S. If this is the first time that you save the current configuration, you will be asked for the filename to save to.

                                                                                      Save configuration to different file

                                                                                      At any time you can save the current configuration to a file of choice by selecting 'Save as...' from the 'File' menu or by pressing CTRL+SHIFT+S. You will be asked for the filename to save to.

                                                                                      Changed configuration upon exit

                                                                                      If the current configuration contains changes since the last time it has been saved, you will be asked whether to save those changes or not before exiting the gui.

                                                                                      Drag and drop config file onto dtangler

                                                                                      You can quickly open a dtangler configuration by just dragging a config file from another application or window onto the dtangler window.

                                                                                      Note that you can only drag&drop a single config file.

                                                                                      Changed configuration upon drag and drop config file onto dtangler

                                                                                      If the current configuration contains changes since the last time it has been saved, you will be asked whether to save those changes or not before another configuration is opened.

                                                                                      Drag and drop input folders onto dtangler

                                                                                      You can quickly add input folders to a configuration by just dragging them from another application or window onto the dtangler window.

                                                                                      Drag and drop non file contents onto dtangler

                                                                                      Dtangler currently only supports drag&drop for input folder(s) or configuration file

                                                                                      DSM interaction

                                                                                      Zoom in

                                                                                      If the DSM that you are currently viewing, contains dependencies on a deeper scope (level), then it is possible to 'zoom in' onto specific parts of the level below the current one, by selecting those items from the DSM.

                                                                                      After making a selection, Zooming in can be done by pressing the 'zoom in' button or by selecting Zoom in onto selection from the View menu.

                                                                                      Zoom in onto specific items

                                                                                      To zoom in onto specific items in the DSM, select those items from the left-side name column of the DSM. Multiple items can be selected.

                                                                                      Zoom in onto dependencies

                                                                                      To zoom in onto dependencies in the DSM, select the cells of those dependencies from the DSM. Either single cells or an area of cells can be selected.

                                                                                      Test zoom out

                                                                                      Currently, 'zoom out' does the same as selecting the scope above the current scope from the 'Scope' combobox. You can 'Zoom out' by pressing the 'Zoom out' button or by selecting 'Zoom out' from the 'View' menu.

                                                                                      Showing shortened item names

                                                                                      By default, the dsm shows the full name of each item.

                                                                                      In many domains and scopes, however, all items inside the DSM tend to differ from eachother ony at the start or end of the name.

                                                                                      To make the DSM more readible, it is possible to hide parts of the name that all items have in common. Only whole words and numbers are hidden from the start and/or end of the item's name.

                                                                                      This can be done by clicking the 'show shortened names' button on the toolbar or selecting 'show shortened names' from the 'View' menu.
